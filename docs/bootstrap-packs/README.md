# Bootstrap Pack Examples

This directory contains starter contracts for utility repos that publish `tusk-bootstrap.json`. Bootstrap packs are how `/tusk-init` turns a user's project intent into concrete starter software, durable project memory, and an actionable first backlog.

Use these files as source templates when creating or updating companion repos:

- `ios-libs/tusk-bootstrap.json` is a schema-v2 example for `gioe/ios-libs`.
- `android-libs/CONTRACT.md` describes the expected future Android pack.
- `web-libs/CONTRACT.md` describes the expected future web pack.
- `backend-libs/CONTRACT.md` describes the expected future backend pack.

The JSON examples are validated by `tests/unit/test_bootstrap_pack_examples.py` with the same validator used by `tusk init-fetch-bootstrap`.

Future repos may be absent from the built-in selector catalog. When a matching optional pack has no configured repo, `tusk init-bootstrap-select` reports it under `skipped_modules` instead of failing project initialization.

## Manifest Contract

A utility repo publishes one root-level `tusk-bootstrap.json`. Legacy task-only manifests are still valid, but new utility repos should use schema v2:

```json
{
  "version": 1,
  "manifest_schema_version": 2,
  "project_type": "web_app",
  "tasks": [],
  "modules": [],
  "manifest_files": []
}
```

Top-level keys:

- `version`: utility manifest version controlled by the utility repo.
- `manifest_schema_version`: set to `2` for module-aware manifests.
- `project_type`: project category this pack serves, such as `ios_app`, `android_app`, `web_app`, or `backend`.
- `tasks`: generic setup tasks for adopting the utility repo.
- `modules`: composable capabilities selected from init intent.
- `manifest_files`: deterministic files or snippets the bootstrap plan can materialize.

## Module Shape

Use modules for capabilities that may or may not apply to every project using the utility repo. Examples include design systems, API clients, navigation shells, observability, persistence, auth, background jobs, or test scaffolding.

Each module should include:

- `id`, `name`, and `description` for plan review.
- `applicability` rules using arrays for `project_types`, `archetypes`, `platforms`, `requires`, and `excludes`.
- optional `files`, `append_operations`, and `manifest_files` style entries for starter code.
- optional `context_atoms`, `pillars`, and `glossary` entries for durable memory.
- optional `tasks` for module-specific setup work.
- optional `verification_hints` describing how a future task should prove the module works.

Applicability should be explainable. A user reviewing the plan should be able to see why a module matched, for example `project_type=ios_app`, `archetype=consumer_ios_app`, `platform=ios`, or `requires=swiftui`.

## Tasks Versus Vertical Slices

Utility packs should keep setup tasks narrow. Good setup tasks install packages, wire shared clients, configure design tokens, add logging, or create test fixtures. They should not pretend to know the whole product roadmap.

First vertical-slice tasks are generated by Tusk from the user's init intent and selected modules. They describe the first useful workflow across behavior, data, integrations, tests, and docs. Pack authors can influence those slices through module metadata, context atoms, verification hints, and quality-priority language, but should not replace them with broad generic tasks.

## Safe Materialization

Use file materialization only for starter assets the utility repo can safely own or update. Every file spec must be safe to run more than once:

- `create_only`: write a new file only when the path does not already exist. Use this for app-owned starter source files and docs.
- `append_if_missing`: append exact content only when it is not already present. Use this for dependency manifests, ignore files, and line-oriented config.
- `marker_block`: replace only the content between explicit `begin_marker` and `end_marker`. Use this for managed sections inside user-owned manifests such as Gradle, Package.swift, or framework config files.

Do not use bootstrap files to overwrite user-authored source. Paths must be relative to the consuming project, and manifests should avoid absolute paths, parent traversal, or ambiguous generated ownership.

Manifest content may use deterministic templates such as `{{ project_name }}` or `{{ init_intent.platforms.0 }}`. Missing template values are conflicts, not empty strings, so a bad manifest fails visibly before it writes misleading code.

## Idempotent Reruns

Assume users will run init more than once. The expected rerun behavior is:

- existing `create_only` files are skipped;
- existing appended snippets are skipped;
- marker blocks are refreshed without touching content outside the markers;
- duplicate setup and vertical-slice tasks are skipped by task insertion;
- existing context atoms, design pillars, and glossary terms are skipped.

This idempotency is part of the public contract. It lets Tusk provide aggressive upfront software while staying safe in partially initialized or manually edited projects.
